For this prototype, we aligned to SIA 416 (areas/levels) and KBOB material codes to enable consistent ecological accounting and reporting across assets. challenge_h2: "Unify IFC attributes and Swiss standards in Power BI without extra work for planners" challenge: | Existing pilot projects store information in varying IFC locations, making it hard to aggregate materials, quantities, and room/level data. Zürich wanted a standards-based report that: (1) reads IFC attributes reliably; (2) classifies materials via KBOB codes; (3) filters by building and floor per SIA 416; and (4) can be maintained by the team without asking planners to remodel data. A lightweight, Power BI-first approach was preferred, with the option to migrate to a database later. solution_h3: "Excel-driven KBOB mapping with embedded model context in Power BI" solution_intro: | Flinker set up a Power BI report that connects IFC element data with an Excel table containing KBOB mappings. Filters for building and floors (SIA 416) let users slice results while keeping the 3D model context available via the embedded IFC visual in Power Bi. Governance stays simple — update the IFC models stored in SharePoint and refresh. For publishing, an iFrame (e.g., in SharePoint or Zürich city publishing portal) keeps access straightforward for wider stakeholders. solution_steps: - "Provide example IFC models to validate attribute placement and end-to-end dataflow." - "Normalize material identification via KBOB codes, including codes appearing within text fields." - "Join IFC quantities/attributes with the Excel KBOB table in a Power BI dataset." - "Add building and floor selectors aligned to SIA 416; keep 3D viewer available alongside visuals." - "Agree on publishing approach (iFrame) and access points (e.g., SharePoint) to minimize authoring-license needs." - "Coordinate storyboard inputs with Build Analytics for competition dashboards." solution_notes: | Start with Excel as the authoritative mapping source to reduce coordination overhead; later, swap to a managed database without changing the reporting logic. results_intro: "Outcomes reported by the user:" results_list: - "Competition dashboards; simple model scopes (e.g., walls, floors, doors, rooms) covered first" - "Clear publishing (e.g., SharePoint)" client_fit_h2: "Zürich’s BIM strategy: data ownership & long-term flexibility" client_fit: |

The Flinker Power BI IFC Visual aligns with AHB’s principles by keeping data in the city’s Microsoft 365 tenant and avoiding lock-in:

  • Data stays in Microsoft 365: Reports, mappings, and files live in SharePoint/OneDrive and Power BI—no external hosting is required
  • Open standards first: Works over IFC, remaining authoring-tool agnostic and portable across projects
  • Flexible governance: Start with Excel/Lists for KBOB mappings; later swap to a database without changing report logic
  • Embed anywhere the city works: Publish via SharePoint/Teams/web using standard M365 security and access control
  • Iterate without vendor lock-in: Models, mappings, and visuals can evolve independently of any single platform
