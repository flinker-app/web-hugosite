The Challenge

How It Works

From policy to compliance in 5 steps

1

Upload Policy-JSON to Site Assets

Define your ISO rules: naming patterns, metadata requirements, approval workflows, and area structure.

2

App validates policy and sets up structure

Libraries created: WIP, Shared, Published, Archive. Fields, views, and versioning configured automatically.

3

Users upload documents

Upload with wrong naming? Blocked with clear guidance. Users see exactly what to fix.

4

Fix and request approval

Correct naming, add required metadata. Approval workflow starts. Approvers notified.

5

Approved = Published

File moves to Published. Audit event logged. Only approved files visible to readers.

Compliant filename structure

PRJ1
Project Code
ORIG
Originator
VOL1
Volume/System
L01
Level/Location
DR
Type (Drawing)
ARC
Discipline
0123
Number
B_Rev03
Status & Revision

Status Codes

  • S0 — Work in progress (WIP)
  • S1 — For coordination (Shared)
  • A — For review/approval (Shared)
  • B — Approved / For construction (Published)

Supported Extensions

  • .ifc — IFC Models
  • .pdf — Documents
  • .dwg — CAD Drawings
  • .xlsx — Spreadsheets

FAQ

Frequently asked questions

All rules are defined in a single Policy-JSON file stored in Site Assets. The JSON schema includes naming patterns (regex), metadata requirements, approval workflows, area structure (WIP/Shared/Published/Archive), and role definitions. Partners can export/import and customize blueprints.

The user receives clear guidance on what's wrong and how to fix it. Depending on the mode (warn/enforce), the file is either flagged with a warning or blocked from proceeding. A compliance issue is logged for tracking.

Files can only move from Shared to Published if they pass all compliance checks: correct naming, required metadata present, and approval granted (if required). V1 enforces this at the UI/process level. Enterprise adds hard enforcement for all upload paths (sync, API, third-party).

Yes. Compliance issues are logged in a SharePoint list (FLK_CDE_ComplianceIssues). Audit events (policy loaded, action attempted, action blocked, approved, rejected, file moved) are logged in FLK_CDE_AuditEvents. Long-term retention integrates with Microsoft Purview.

Partners can create, export, import, and version blueprints (JSON). Build a "DB InfraGO profile" or "UK ISO 19650 profile" once, then deploy to multiple clients. Customizing without code — change the JSON, redeploy.

V1 (Pro): UI/process gate enforcement. Validates at release actions (submit, approve, publish). Users can still upload via sync/API but files won't pass the release gate if non-compliant.

Enterprise: Hard enforcement on all upload paths. Blocks non-compliant files at ingress. Plus transmittals suite, advanced reporting, and priority support.
